---
family: SC
family_name: System and Communications Protection
id: SC-28
name: Protection of Information at Rest
---

question: "Is sensitive data encrypted at rest — and is that encryption actually enforced, not just available?"

description: >
  The information system protects the confidentiality and integrity of information at rest.
  Encryption at rest means that data stored on disk, in a database, in object storage, or
  in a backup is encrypted such that physical access to the storage medium does not yield
  readable data. The common failure is encryption being available but not enforced — a
  setting that must be opted into rather than a default that must be opted out of. In K8s,
  this means etcd encryption for Secrets, PVC encryption for persistent volumes, and
  encrypted EBS/Azure Disk volumes. In cloud, this means KMS-managed encryption on every
  storage service, not just the ones that are obvious.

enhancements:
  - id: SC-28(1)
    name: Cryptographic Protection
    description: >
      The information system implements cryptographic mechanisms to prevent unauthorized
      disclosure and modification of information at rest. Encryption must be implemented
      using approved algorithms (AES-256) with keys managed through a dedicated key
      management service — not platform-default keys shared across all customers.
      Customer-managed keys (CMKs) are required where data sensitivity or compliance
      mandates key custody evidence.

HITRUST_map:
  - "10.f — Policy on the Use of Cryptographic Controls"
  - "10.g — Key Management"
  - "09.ab — Monitoring System Use"

evidence:
  what_to_look_for:
    - etcd encryption at rest configuration (EncryptionConfiguration resource in K8s — verify Secrets are encrypted, not just base64)
    - Persistent volume encryption (EBS encryption, Azure Disk encryption, GCE PD encryption) enabled by default
    - Object storage encryption (S3 SSE-KMS, Azure Blob encryption) enforced via bucket/account policy — not optional
    - Database encryption at rest (RDS storage encryption, Azure SQL TDE) enabled on all instances
    - Backup encryption (automated snapshot encryption, backup vault encryption)
    - Customer-managed key (CMK) configuration where required — verify key policy and rotation
  ask_for:
    - "Show me your K8s EncryptionConfiguration — are Secrets encrypted in etcd or are they stored as base64 plaintext?"
    - "Show me your S3 bucket default encryption settings — is SSE-KMS enforced, and does a bucket policy deny unencrypted PutObject?"
    - "Show me that all EBS volumes attached to your EC2 and K8s nodes are encrypted — are there any unencrypted volumes?"
    - "Show me your database encryption configuration — is storage encryption enabled on all RDS instances, and is it using a CMK?"
  tools:
    generic:
      - kubectl (`kubectl get secret <name> -o jsonpath='{.data}'` — base64 output means etcd encryption may not be active; verify via API server config)
      - kube-bench (checks etcd encryption configuration as part of CIS benchmark)
      - Trivy (misconfiguration scan — detects unencrypted K8s volumes and cloud storage)
      - Kubescape (etcd encryption and secret hygiene checks)
    aws:
      - AWS Config (rule: encrypted-volumes, rds-storage-encrypted, s3-bucket-server-side-encryption-enabled)
      - AWS KMS (verify CMK usage on S3, RDS, EBS, EFS, DynamoDB, SQS)
      - S3 bucket policy (deny `s3:PutObject` without `x-amz-server-side-encryption` header)
      - AWS Security Hub (encryption findings across all storage services)
      - AWS Macie (detect unencrypted sensitive data in S3)
    microsoft:
      - Azure Policy (deny storage accounts without encryption at rest — built-in policy)
      - Azure Disk Encryption (ADE — OS and data disk encryption with customer-managed keys)
      - Azure Storage Service Encryption (enforce with CMK in Key Vault)
      - Microsoft Defender for Cloud (encryption recommendations across storage services)
      - Azure SQL Transparent Data Encryption (TDE with CMK — verify key vault integration)

failure_to_implement:
  - K8s Secrets stored as base64 in etcd without EncryptionConfiguration — any etcd backup yields all cluster secrets in plaintext.
  - S3 bucket encryption set to SSE-S3 (AWS-managed key) but no bucket policy prevents unencrypted uploads — objects uploaded without encryption header are stored in plaintext.
  - EBS volume attached to a worker node is unencrypted — physical or snapshot access to the volume exposes all data on it.
  - Database backup snapshots are not encrypted — a snapshot shared or exfiltrated exposes the full dataset.
  - Encryption is enabled using default platform-managed keys — no CMK means no custody evidence for key access, failing FedRAMP SC-28(1).
