---
family: SC
family_name: System and Communications Protection
id: SC-13
name: Cryptographic Protection
---

question: "Are we using approved, strong cryptographic algorithms — and have weak or deprecated algorithms been eliminated?"

description: >
  The information system implements approved cryptography in accordance with applicable
  federal laws, executive orders, directives, regulations, policies, standards, and guidance.
  Approved cryptography means FIPS 140-2 or 140-3 validated implementations for federal
  systems, and NIST-approved algorithms (AES-256, SHA-256/384, RSA-2048+, ECDSA P-256+)
  for all environments. SC-13 is not about whether encryption is in use — SC-8 and SC-28
  cover that. SC-13 is about whether the cryptography being used is algorithmically sound
  and from a validated implementation. MD5, SHA-1, DES, 3DES, RC4, and 1024-bit RSA are
  all deprecated and must be absent from production systems.

enhancements:
  - id: SC-13(1)
    name: FIPS-Validated Cryptography
    description: >
      The information system implements FIPS-validated cryptography to protect classified
      information and controlled unclassified information (CUI). For FedRAMP and DoD
      environments, FIPS 140-2 or 140-3 validation is mandatory — not optional. FIPS mode
      must be enabled at the OS and application runtime level, not just at the library level.

HITRUST_map:

evidence:
  what_to_look_for:
    - Approved algorithm list documented in security policy (AES-256, SHA-256+, RSA-2048+, ECDSA P-256+)
    - TLS configuration showing deprecated protocols and weak cipher suites are disabled
    - Application cryptography audit — hash functions, encryption libraries, and algorithm choices in code
    - FIPS 140-2/3 validation certificates for cryptographic modules in use (for federal environments)
    - Evidence that MD5, SHA-1, DES, RC4, and RSA < 2048-bit are not in use anywhere in the system
  ask_for:
    - "Show me your approved algorithm list — what ciphers, hash functions, and key lengths are permitted, and who approved the list?"
    - "Show me your TLS cipher suite configuration — are MD5, RC4, or NULL cipher suites explicitly disabled?"
    - "Show me how your application generates password hashes — is it using bcrypt, Argon2, or PBKDF2? Is MD5 or SHA-1 used anywhere?"
    - "For FedRAMP: show me FIPS 140-2 validation certificates for the cryptographic modules your system relies on."
  tools:
    generic:
      - testssl.sh (cipher suite audit — identifies weak, deprecated, and NULL cipher suites)
      - openssl ciphers -v (enumerate supported cipher suites for a given TLS configuration)
      - Semgrep / Bandit (SAST rules detecting use of weak hash functions in application code)
      - OWASP Cryptographic Storage Cheat Sheet (algorithm selection reference)
      - grype / Trivy (detect libraries with known weak cryptography implementations)
    aws:
      - AWS Config (rule: alb-http-drop-invalid-header-enabled — supplemental TLS hygiene)
      - AWS Security Hub (findings related to weak encryption configurations)
      - AWS KMS (enforces FIPS 140-2 validated key operations — all KMS API calls use validated module)
      - ACM (enforces modern TLS cipher suites on managed certificates)
    microsoft:
      - Azure Key Vault (FIPS 140-2 validated operations in standard tier; Level 3 in Premium/HSM)
      - Azure App Service TLS settings (minimum TLS version and cipher suite configuration)
      - Microsoft Defender for Cloud (recommendations for weak TLS configurations)
      - Azure Policy (deny creation of resources with deprecated TLS versions)

failure_to_implement:
  - SHA-1 still in use for code signing — deprecated since 2017, collisions are computationally feasible.
  - Application password hashes stored as MD5 — a database breach immediately yields cracked passwords via rainbow tables.
  - TLS 1.0/1.1 or RC4 cipher suites still accepted — environment appears encrypted but is vulnerable to known attacks.
  - FedRAMP system using non-FIPS-validated cryptographic libraries — fails SC-13 and triggers a Plan of Action and Milestones (POA&M) finding.
  - RSA-1024 certificates still in rotation — below NIST minimum key length since 2011, considered computationally breakable.
