question: "Do audit records contain enough information to reconstruct what happened, who did it, and when?"

description: >
  The information system generates audit records that contain sufficient information to
  establish: what type of event occurred, when it occurred, where it occurred, the source
  of the event, the outcome of the event, and the identity of any associated individuals
  or subjects. A log entry that captures only a timestamp and an action is not sufficient
  — attribution (who), context (where and from what), and outcome (success/failure) are
  all required. AU-3 defines the minimum field set; AU-12 ensures those records are actually
  generated.

enhancements:
  - id: AU-3(1)
    name: Additional Audit Information
    description: >
      The information system generates audit records containing organization-defined
      additional information beyond the baseline fields. In K8s environments this typically
      includes namespace, resource name, requestURI, userAgent, and sourceIPs. In cloud
      environments this includes account ID, region, and request parameters relevant to the action.
  - id: AU-3(2)
    name: Centralized Management of Planned Audit Record Content
    description: >
      The organization provides centralized management and configuration of the content to
      be captured in audit records generated by defined system components. Prevents individual
      teams from independently modifying logging verbosity in ways that create gaps.

evidence:
  what_to_look_for:
    - Sample audit log entries showing all required fields (event type, timestamp, source, outcome, subject identity)
    - K8s audit log entries at Request or RequestResponse level for sensitive operations — verify field completeness
    - Structured log format documentation (JSON schema or field list) for each logging system
    - Evidence that failed events are captured, not only successes
    - Log field mapping document showing how each required field is populated across different system components
  ask_for:
    - "Show me a sample audit log entry for a privileged operation — does it include who, what, when, from where, and success/failure?"
    - "Show me a sample K8s audit log entry for a secret access event — does it include the requesting user, source IP, and response code?"
    - "Show me how application authentication failures are logged — are failed attempts captured with the same field completeness as successes?"
    - "Show me a CloudTrail or Azure Activity Log entry for a policy change — does it include the request parameters, not just the action name?"
  tools:
    generic:
      - K8s audit log (verify fields: user.username, user.groups, sourceIPs, verb, resource, namespace, responseStatus.code)
      - jq (parse and validate log field completeness against required schema)
      - Falco (verify alert output includes all required attribution fields)
      - auditd log format (check ausearch output for required fields)
    aws:
      - CloudTrail (verify log entries include: eventTime, userIdentity, sourceIPAddress, eventName, requestParameters, responseElements, errorCode)
      - AWS Config (configuration change records — verify resource identity and change delta captured)
      - VPC Flow Logs (verify flow record format includes src/dst IP, port, protocol, bytes, action)
    microsoft:
      - Azure Activity Log (verify operationName, caller, correlationId, resourceId, status)
      - Entra ID Sign-in Logs (verify fields: userPrincipalName, ipAddress, appDisplayName, conditionalAccessStatus, riskState)
      - Microsoft Sentinel (normalize log fields to ASIM schema — verify field coverage)

failure_to_implement:
  - Log entries capture event type and timestamp only — no source IP or user identity makes attribution impossible post-incident.
  - Failed authentication attempts logged without the attempted username — cannot identify credential stuffing targets.
  - K8s audit policy set to Metadata level for secret access — request body not captured, actual secret name unknown.
  - Cloud API calls logged with action name but no request parameters — cannot determine what resource was affected.
  - Incident investigation stalls because logs exist but lack the fields needed to reconstruct the attack timeline.
