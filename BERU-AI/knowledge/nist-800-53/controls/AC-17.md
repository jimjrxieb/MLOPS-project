question: "Is remote access controlled, encrypted, and logged?"

description: >
  The organization establishes and documents usage restrictions, configuration requirements,
  connection requirements, and implementation guidance for each type of remote access allowed.
  Remote access is authorized before allowing connections and monitored in accordance with
  audit controls. In cloud-native and K8s environments, remote access includes kubectl from
  off-network machines, bastion/jump host sessions, API server exposure, and any management
  plane access that crosses a network boundary.

enhancements:
  - id: AC-17(1)
    name: Monitoring and Control
    description: >
      The organization employs automated mechanisms to monitor and control remote access sessions.
      Passive documentation is insufficient — active monitoring with alerting on anomalous
      session characteristics (unusual hours, source IP, volume) is required.
  - id: AC-17(2)
    name: Protection of Confidentiality and Integrity Using Encryption
    description: >
      The information system implements cryptographic mechanisms to protect the confidentiality
      and integrity of remote access sessions. All remote management connections must be
      encrypted in transit — no plaintext protocols (Telnet, HTTP, unencrypted RDP).
  - id: AC-17(3)
    name: Managed Access Control Points
    description: >
      The organization routes remote access via managed network access control points.
      Direct node access or unapproved jump paths are prohibited — all remote sessions
      must pass through a controlled chokepoint (VPN, bastion, Session Manager).
  - id: AC-17(4)
    name: Privileged Commands and Access
    description: >
      The organization authorizes the execution of privileged commands and access to
      security-relevant information via remote access only for documented operational needs.
      Ad-hoc remote admin without a documented justification is a policy violation.

evidence:
  what_to_look_for:
    - VPN or bastion configuration showing MFA enforcement for remote sessions
    - K8s API server access policy (network ACL, kubeconfig restrictions, OIDC/mTLS enforcement)
    - Remote session logs showing source IP, user, duration, and commands executed
    - Evidence that direct SSH to cluster nodes is disabled or gated through Session Manager/bastion
    - Encryption-in-transit configuration for all remote management protocols
  ask_for:
    - "Show me how kubectl access to the cluster is authenticated from outside the corporate network — is MFA enforced?"
    - "Show me if direct SSH to worker nodes is possible. If so, show me the controls — who can, logged where?"
    - "Show me the remote access policy document — does it list approved connection methods, session controls, and prohibited protocols?"
    - "Show me a remote session log entry — what fields are captured and where are they retained?"
  tools:
    generic:
      - kubectl config view (verify cluster API server endpoint and auth method)
      - ssh audit (verify SSH config — ciphers, key types, password auth disabled)
      - WireGuard / OpenVPN configuration review
    aws:
      - AWS Systems Manager Session Manager (replace direct SSH — sessions logged to S3/CloudWatch)
      - AWS Client VPN (MFA-enforced remote access logs)
      - CloudTrail (AssumeRole events from external IPs, console logins)
      - VPC Security Groups (restrict API server access to known CIDRs)
    microsoft:
      - Azure Bastion (browser-based SSH/RDP without public IP exposure)
      - Entra ID Conditional Access (enforce MFA + compliant device for remote access)
      - Microsoft Defender for Endpoint (remote session anomaly detection)
      - Azure Monitor / Log Analytics (session activity logs)

failure_to_implement:
  - Plaintext protocols (Telnet, HTTP) used for remote management expose credentials to network interception.
  - Direct SSH to worker nodes without bastion or session logging leaves no audit trail of admin actions.
  - kubectl configured with long-lived static credentials that are not rotated or audited.
  - MFA not enforced for VPN — a single stolen credential grants full remote access.
  - No alerting on remote access from anomalous source IPs or at unusual hours misses account takeover.
