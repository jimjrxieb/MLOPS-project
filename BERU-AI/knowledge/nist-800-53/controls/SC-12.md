id: SC-12
name: Cryptographic Key Establishment and Management
---

question: "Are cryptographic keys generated, distributed, stored, rotated, and revoked under a defined policy?"

description: >
  The organization establishes and manages cryptographic keys for required cryptography
  employed within the system in accordance with key management requirements. Cryptography
  is only as strong as the key management behind it. A well-implemented cipher with a
  hardcoded, never-rotated, or improperly stored key provides weaker security than a
  modest cipher with rigorous key management. SC-12 governs the full lifecycle: generation
  using approved sources of randomness, secure distribution without exposure, storage
  with access controls, rotation on a defined schedule, and revocation when compromise
  is suspected or confirmed.

enhancements:
  - id: SC-12(1)
    name: Availability
    description: >
      The organization maintains availability of information in the event of the loss of
      cryptographic keys. Key escrow, key backup, and recovery procedures ensure that
      losing a key does not mean losing access to the data it protects. Recovery must
      be tested — an untested recovery procedure is not a recovery procedure.
  - id: SC-12(2)
    name: Symmetric Keys
    description: >
      The organization produces, controls, and distributes symmetric cryptographic keys
      using approved key management technology and processes. Symmetric keys must be
      generated using FIPS 140-validated modules or equivalents and distributed only
      through encrypted channels.
  - id: SC-12(3)
    name: Asymmetric Keys
    description: >
      The organization produces, controls, and distributes asymmetric cryptographic keys
      using approved PKI infrastructure or approved key management technology and processes.
      Private keys must never leave the HSM or key management service in plaintext.

HITRUST_map:
  - "10.f — Policy on the Use of Cryptographic Controls"
  - "10.g — Key Management"

evidence:
  what_to_look_for:
    - Key management policy documenting lifecycle phases, rotation schedules, and approved algorithms
    - Key storage in a dedicated secrets management or KMS system — not in files, environment variables, or source code
    - Rotation records showing keys are rotated on schedule and after personnel changes or suspected compromise
    - Key access audit log — who accessed which key, when, and for what purpose
    - Key recovery procedure documentation and evidence of a tested recovery exercise
    - HSM or FIPS 140-2 validated module used for key generation where required
  ask_for:
    - "Show me your key management policy — what is the rotation schedule for each key type (TLS certs, encryption keys, signing keys)?"
    - "Show me where your encryption keys are stored — are they in AWS KMS, Azure Key Vault, or HashiCorp Vault? Are any stored outside a KMS?"
    - "Show me your key rotation records — when were your production data encryption keys last rotated?"
    - "Show me your key recovery procedure and evidence that it has been tested — what happens if the primary key is lost?"
  tools:
    generic:
      - HashiCorp Vault (key lifecycle management, automatic rotation, audit log)
      - cert-manager (TLS certificate lifecycle — automated renewal via Let's Encrypt or private CA)
      - PKCS#11 / HSM integration (hardware-backed key generation and storage)
      - Sealed Secrets (K8s-native encrypted secret storage using asymmetric keys)
    aws:
      - AWS KMS (centralized key management — automatic annual rotation, key policy, CloudTrail integration)
      - AWS CloudHSM (FIPS 140-2 Level 3 validated HSM for highest-assurance key storage)
      - AWS Config (rule: cmk-backing-key-rotation-enabled)
      - CloudTrail (Decrypt, GenerateDataKey, ScheduleKeyDeletion events)
      - AWS Secrets Manager (secret rotation with Lambda-based rotation functions)
    microsoft:
      - Azure Key Vault (key, secret, and certificate management — FIPS 140-2 Level 2 software HSM)
      - Azure Dedicated HSM (FIPS 140-2 Level 3 for highest-assurance requirements)
      - Azure Key Vault key rotation policy (automated rotation with expiry notification)
      - Azure Monitor (Key Vault diagnostic logs — key access and rotation events)

failure_to_implement:
  - Encryption keys stored alongside the data they protect — compromise of the data store also yields the key.
  - No rotation schedule — keys in use for years with no record of when they were created or who has ever accessed them.
  - Private keys exported from KMS and stored on developer workstations "for convenience."
  - Key recovery never tested — a key loss scenario triggers hours of downtime because the recovery procedure does not actually work.
  - Different teams manage different keys with no unified policy — rotation cadences, storage locations, and access controls are inconsistent.
