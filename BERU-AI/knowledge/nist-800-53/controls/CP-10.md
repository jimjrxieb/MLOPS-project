question: "Can the system be recovered to a known-good state after a disruption — within the defined RTO and with verified integrity?"

description: >
  The organization provides for the recovery and reconstitution of the system to a known
  state within organization-defined time periods consistent with recovery time and recovery
  point objectives after a disruption, compromise, or failure. Recovery is the operational
  execution of what CP-9 (backup) makes possible. The distinction matters: CP-9 is having
  the data; CP-10 is successfully using it to restore a functioning system within the
  required timeframe. Reconstitution means the restored system is verified to be in a
  known-good, secure state — not just running, but running with controls intact, no
  attacker persistence remaining, and configuration matching the approved baseline.

enhancements:
  - id: CP-10(2)
    name: Transaction Recovery
    description: >
      The information system implements transaction recovery for systems that are
      transaction-based. Database transaction logs and point-in-time recovery ensure
      that a failure mid-transaction does not result in partial data corruption.
      RPO at the transaction level rather than the backup-interval level.
  - id: CP-10(4)
    name: Restore Within Time Period
    description: >
      The organization provides the capability to restore system components within
      organization-defined restoration time periods from configuration-controlled and
      integrity-protected information representing a known, operational state. The
      restoration time period is the operational RTO — it must be defined, documented,
      and verified through testing. An RTO that has never been tested is a guess,
      not a commitment.

evidence:
  what_to_look_for:
    - Defined and documented RTO and RPO for each system component or criticality tier
    - Recovery runbook with step-by-step instructions for restoring the system from backup
    - Recovery test records showing RTO was achieved during a planned exercise
    - Post-recovery integrity verification procedure — steps to confirm the restored system matches the approved baseline
    - Evidence that recovered system is security-verified before returning to production (vulnerability scan, configuration check)
    - Reconstitution checklist confirming no attacker persistence (changed credentials, verified clean image, reviewed audit logs)
  ask_for:
    - "Show me your defined RTO and RPO — for each system tier, what are the targets and how were they determined?"
    - "Show me your recovery runbook — if I had to restore production K8s from scratch at 2 AM, what are the exact steps?"
    - "Show me your most recent recovery test results — did you meet your RTO, and what issues were encountered?"
    - "Show me your post-recovery integrity verification process — how do you confirm the restored system is clean and configuration-compliant before going back to production?"
  tools:
    generic:
      - Velero (K8s cluster restore — namespace, PVC, and workload recovery from backup)
      - Terraform (infrastructure reconstitution — recreate cloud resources from IaC in a new region)
      - Helm (application re-deployment from versioned chart — known-good application state)
      - Ansible (node configuration restoration to known-good baseline)
      - etcd restore (K8s control plane recovery from etcd snapshot)
    aws:
      - AWS Backup restore (point-in-time recovery for EC2, RDS, EFS, DynamoDB, EKS)
      - AWS Elastic Disaster Recovery (continuous replication with sub-hour RTO)
      - CloudFormation StackSets (infrastructure reconstitution from template in alternate region)
      - RDS Point-in-Time Recovery (database restoration to specific transaction timestamp)
      - AWS Systems Manager Automation (recovery runbook automation — codified restore steps)
    microsoft:
      - Azure Site Recovery (orchestrated failover and recovery with defined RTO targets)
      - Azure Backup restore (VM, database, and file recovery with restore point selection)
      - Azure Kubernetes Service (cluster recreation from ARM template and Velero backup)
      - Azure SQL point-in-time restore (database recovery to specific transaction point)

failure_to_implement:
  - RTO is defined as 4 hours in the contingency plan but has never been tested — actual recovery takes 18 hours during an incident.
  - Recovery runbook exists but references outdated architecture — the restore steps fail because the system has changed since the runbook was written.
  - System is restored from backup but post-recovery integrity check is skipped — attacker persistence (a backdoor added before the incident) survives the recovery.
  - Recovery is successful but the restored system runs without security controls (Kyverno, Falco not re-deployed) until someone notices days later.
  - No recovery capability for the K8s control plane — etcd backup exists but no one has documented or practiced the restore procedure.
