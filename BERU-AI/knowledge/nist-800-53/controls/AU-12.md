question: "Is the system actually generating audit records for every event defined in AU-2?"

description: >
  The information system provides audit record generation capability for the events defined
  in AU-2; allows authorized personnel to select which events are to be audited by specific
  system components; and generates audit records for defined events with the content defined
  in AU-3. AU-12 is the execution control — AU-2 says what to log, AU-3 says what fields
  to include, AU-12 verifies the system is actually doing it. Coverage gaps here mean that
  an entire class of events (secret access, privilege escalation, config changes) can occur
  with no audit record generated.

enhancements:
  - id: AU-12(1)
    name: System-Wide and Time-Correlated Audit Trail
    description: >
      The information system compiles audit records from multiple components into a system-wide,
      time-correlated audit trail. All components must emit logs synchronized to a common
      time source (NTP) so records from different systems can be sequenced accurately during
      incident reconstruction. A one-minute clock skew across components can make a causal
      chain appear as simultaneous or reversed events.
  - id: AU-12(3)
    name: Changes by Authorized Individuals
    description: >
      The information system provides the capability for authorized individuals to change
      the auditing to be performed within organization-defined time thresholds. Authorized
      security personnel must be able to increase audit verbosity (e.g., enable RequestResponse
      logging for a specific user during an investigation) without waiting for a change
      management cycle.

evidence:
  what_to_look_for:
    - Audit log coverage verification — evidence that each event category in AU-2 has corresponding log entries
    - NTP configuration showing all system components synchronized to a common time source
    - Log pipeline health monitoring — alerting on gaps, drops, or delays in log generation
    - K8s audit policy deployed to all API server instances (not just control plane leader)
    - Cloud logging service enabled across all regions, not just primary region
    - Proof that audit record generation cannot be disabled by non-security personnel
  ask_for:
    - "Show me how you verify that all events in your AU-2 event list are actually being captured — do you have a coverage test or spot-check process?"
    - "Show me your NTP configuration across cluster nodes — are all nodes synchronized and what's the maximum permitted skew?"
    - "Show me your log pipeline monitoring — what alerts fire if a component stops generating logs?"
    - "Show me that CloudTrail or Azure Monitor is enabled in every region your workloads run in, not just your primary region."
  tools:
    generic:
      - timedatectl / chronyc (verify NTP sync status on all nodes)
      - Fluent Bit metrics (monitor log pipeline throughput — alert on drop rate)
      - K8s audit log coverage test (generate a known event, verify it appears in the audit log)
      - Falco (verify Falco is running on all nodes and generating events)
    aws:
      - CloudTrail multi-region trail (verify all regions covered — not just us-east-1)
      - AWS Config (rule: cloud-trail-enabled, multi-region-cloud-trail-enabled)
      - CloudWatch Logs metric filters (verify audit log stream is active — alert on silence)
      - AWS Config recorder (verify recording is active for all resource types, all regions)
    microsoft:
      - Azure Monitor Diagnostic Settings (verify all resource types in all regions forward logs)
      - Azure Policy (deny disabling of diagnostic settings on production resources)
      - Microsoft Defender for Cloud (coverage gaps — identifies resources not sending logs)
      - Azure Arc (extend audit record generation to hybrid/on-prem nodes)

failure_to_implement:
  - K8s audit policy deployed to control plane but worker node logs are not captured — workload-level events have no audit trail.
  - CloudTrail enabled in primary region only — attacker pivots to a secondary region to operate without audit coverage.
  - Clock skew of several minutes between nodes — incident timeline reconstruction is ambiguous; logs cannot establish causality.
  - Log pipeline silently drops 15% of events under high load — audit coverage appears complete but has systematic gaps.
  - Audit log generation disabled by a misconfiguration pushed through an automated pipeline — goes undetected for days.
