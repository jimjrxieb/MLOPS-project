# .gitleaksignore — Why each entry is here

The 16 fingerprints in `.gitleaksignore` are all in artifacts where secret-shaped
strings are intentional content, not real secrets:

| Path pattern | Why allowlisted |
|---|---|
| `4-eval-clarify/2-test-data/training-data/faulty-examples/**` | Deliberately-bad teaching fixtures — files literally named `*-FAULTY*` whose purpose is to demonstrate what an exposed-secret config looks like (the lint job is supposed to flag them, not gitleaks at our layer). |
| `4-eval-clarify/2-test-data/training-data/deployments/training/broken-app/**` | Same — "broken-app" teaching example. |
| `2-rag-ingestion/02-preperation-factory/stages/sanitize_npc.py` | The sanitizer's own regex pattern — a literal string in code that detects credentials, not a credential. |
| `JADE-AI/tasks/day3.md` | Curriculum task file — the secret-shaped string is an example used in a teaching exercise. |
| `BERU-AI/training-data/chatml-examples/beru-training-examples.jsonl` `1-local-pipeline/01-raw-data-lake/beru_training_*.jsonl` `1-local-pipeline/06-eval-holdout/eval_set.jsonl` | Training/eval corpora containing the redacted Stripe placeholder `sk_live_REDACTED_PLACEHOLDER_NOT_REAL` and similar — pattern still matches gitleaks's `generic-api-key` rule despite the obvious `REDACTED_PLACEHOLDER` marker. The strings are inert; the model trains on the pattern, not the value. |

**New secret-shaped strings will still fail CI** — gitleaks scans the diff, finds the new fingerprint, doesn't see it in `.gitleaksignore`, exits 1. To accept a new one: rerun gitleaks locally, capture the fingerprint from the report, add it here with a rationale.
