## What broke and why

The `build` job failed at "Set up job" with:
```
Unable to resolve action `aquasecurity/setup-trivy@v0.2.2`
```

Root cause: `aquasecurity/trivy-action@v0.29.0` uses `setup-trivy@v0.2.2` internally as a composite action step. That tag was deleted from the upstream repo. Our CI was pinned to a specific commit hash of `trivy-action`, so there was no automatic update — the failure surfaced when GitHub tried to resolve the sub-action at runtime.

## Fix sequence

### Commit 1 — replace trivy-action with direct install

`trivy-action` composite dependency broke. Replaced with direct `curl` install, same pattern as the gitleaks step. Eliminates the composite action dependency entirely.

```yaml
# Before
uses: aquasecurity/trivy-action@18f2510...
with:
  image-ref: 'beru-api:ci'
  severity: 'HIGH,CRITICAL'
  exit-code: '1'

# After
run: |
  TRIVY_VERSION="0.70.0"
  curl -sSfL -o trivy.tgz \
    "https://github.com/aquasecurity/trivy/releases/download/v${TRIVY_VERSION}/trivy_${TRIVY_VERSION}_Linux-64bit.tar.gz"
  tar xzf trivy.tgz trivy
  ./trivy image --severity HIGH,CRITICAL --exit-code 1 ...
```

**Result:** Trivy now runs. Finds 4 real CVEs. Build still fails.

### Commit 4 — identify false positive root cause, add to .trivyignore

Debugged inside the built image:
```bash
docker run --rm --entrypoint find beru-api:test-local /usr -name "jaraco.context*" -name "*.dist-info"
# /usr/local/lib/python3.11/site-packages/setuptools/_vendor/jaraco.context-5.3.0.dist-info
# /usr/local/lib/python3.11/site-packages/setuptools/_vendor/wheel-0.45.1.dist-info
# /usr/local/lib/python3.11/site-packages/wheel-0.47.0.dist-info
```

Trivy scans the entire filesystem including `setuptools/_vendor/`. Setuptools bundles its own private vendored copies of wheel and jaraco.context for internal build operations. These are **not importable by application code**. The actual runtime packages are at patched versions.

Fix: added CVE-2026-24049 and CVE-2026-23949 to `.trivyignore` with documented root cause. Verified locally (`trivy image ... ; echo "Exit: $?"` → `Exit: 0`) before pushing.

**Result:** All jobs green. ✓

## Lessons

1. **Pin actions to commit SHAs** but be aware that composite actions can still pull in tagged sub-actions at runtime — those tags can be deleted independently of the parent action.

2. **`gh run view <run-id> --log-failed`** is the right command. It filters to only failed steps. For artifact-based output (gitleaks-report.json, trivy table), download with `gh run download`.

3. **Trivy scans `setuptools/_vendor/`** — this is a known false-positive source. Any package setuptools vendors internally (wheel, jaraco.context, etc.) will show up with the version setuptools was built with, not the version you install separately. Document in `.trivyignore` with the explanation.

4. **Local verification before push**: `trivy image --ignorefile .trivyignore ... ; echo "Exit: $?"` — run this locally first. If it exits 0, you can push with confidence.
